## 1. ビジョン（一言）

レベルアップで得た **スキルポイント（SP）** を消費し、**ジャンル別のツリー**上の **パーク（ノード）** を順に開放して、数値強化と**固有アビリティ**の両方で釣り体験を深める。

---

## 2. ユーザー体験（想定フロー）

1. 釣り・売却などで経験値が溜まりレベルアップする。
2. レベルアップごとに **SP を 3 獲得**（未使用分は貯める）。
3. Book UI など既存 UI パターンに沿った **スキル画面** を開く。
4. ツリー（タブまたはページ）を選び、**前提パークを満たしたノード**だけ SP を支払って解放。
5. 解放内容が即時反映（ファイトゲージ、レア補正、新しい受動効果など）。

---

## 3. コア要件（いただいたアイデアの整理）

| 項目 | 内容 |
|------|------|
| 複数ツリー | ジャンルが異なる **複数ライン**（例: ファイト／探索・経済／図鑑・幸運 など）。同時に全部を伸ばすより、**ビルドの方向性**が付く想定。 |
| 取得の順序 | **依存関係付きのパーク**。単線のみ（一本道）か、途中で分岐・合流する **DAG（有向非巡回）** かは要決定。 |
| ノード種別 | **純パラメータ**（% や固定値）と **アビリティ**（条件付き発動・新挙動・UI拡張など）の **2系統（以上）**。 |
| コスト | 解放に **SP 消費**。ノードごとにコストが異なるか、階層で一律かは要決定。 |
| SP の入手 | **主にレベルアップ**。実績報酬や将来のクエストからの付与は **任意（将来拡張）** として仕様に書くか要決定。 |

## 4. 既存システムとの関係（要すり合わせ）

本プロジェクトには既に **レベル** と **レベルボーナス**（4-2）がある。

- **案A（積み上げ）**: レベルボーナスは「基礎成長」、スキルツリーは「特化・ビルド」。数値が二重に乗るため **バランス設計が必須**。
- **案B（役割分担）**: 今後のレベルボーナス伸びを抑え、成長の主軸をツリーに寄せる。
- **案C（タグ分離）**: レベルボーナスは「下限保障」、ツリーは「上限突破・特殊効果のみ」と役割を分ける。

仕様書段階で **どれを採用するか明記**する。

## 5. データモデル案（ドラフト）

確定ではない。実装時に TypeScript 化する。

```text
SkillTreeId        // 例: "combat" | "economy" | "collection" | "luck"
SkillNodeId        // ツリー内一意

SkillNode {
  id, treeId, name, description,
  costSp: number,
  requires: SkillNodeId[],     // 前提パーク（全て解放済みであること）
  effects: Effect[]             // パラメータ加算 / アビリティID 参照 など
}

PlayerData 追記案:
  unlockedSkillNodes: Set<SkillNodeId>  // または string[]
  skillPoints: number                   // 未使用 SP（レベルアップで加算）
  // 既存セーブとのマイグレーション方針を仕様に含める
```

**永続化**: `savePlayerData` / `loadPlayerData` の拡張と、旧セーブに SP・解放集合が無い場合の初期化ルールが必要。

### 10.3 ツリー構造の補足（設計ルール）

- **グラフ形式**: ノードは `requires: SkillNodeId[]`（AND 前提）で表現できる **DAG** とする。合流ノード（複数ルートの統合）を許可する。
- **フェーズの目安**（各ツリー内のざっくり区分。実装時はノード定義で位置づける）:
  - **序盤**: 比較的 **線形に近い** 導入部。選択の負担を抑え、ツリーのテーマを理解させる。
  - **中盤**: **左右（複数）分岐**を置き、ビルドの差を付ける主戦場とする。
  - **終盤（上位）**: **浅い投資では到達しない**ように、前提チェーンを長くする・合流を要求する・コストを高くするなどで **「そのツリーを極めた人向け」** に寄せる。
- **解釈**: 「中盤までに分岐」とは、**新たな左右分岐の追加は中盤レイヤーまで**とし、終盤は分岐の「再拡散」より **収束・集約・キャップストーン** を優先するイメージ（データ設計で具体ノードに落とす）。

### 10.4 レベルボーナスとスキルツリー（確定方針）

- **採用**: 提案書 **セクション4** の **案C（役割分離）** に相当する。
- **意図**: レベルアップは「誰でも少しずつ良くなる**土台**」に留め、**数値・体験の主な伸長はスキルツリー**で行う。
- **現状コードとの対応**（2026-04-03 時点）: レベル連動は `getLevelBarRangeBonus`（バー判定幅）と `getLevelGaugeSpeedBonus`（ゲージ速度）の **2 系統のみ**。案Cに沿うため、今後の調整では次を想定する。
  - 上記レベルボーナスは **弱め・上限付き**に抑える、または **伸び率を下げる** など、**スキル側と二重に盛れない**ようバランスする。
  - ステータスに載らない強化（経済・図鑑・QoL など）は **`special` ツリー**と **`abilityId`** で扱う（10.7.2）。
- **実装タイミング**: スキル効果を接続する際に、既存レベル式の見直しとセットで行う（仕様として「レベルは薄く、スキルで厚く」を満たすこと）。

### 10.5 スキルポイント（SP）の獲得

- **付与タイミング**: プレイヤーが **レベルが上がったとき**（`level` が増加した区間ごと）。1 回のレベルアップで複数レベル跳ねた場合は、**上がったレベル数 × 3 SP** とする（例: Lv5→8 なら +9 SP）。
- **付与量**: **固定 3 SP / レベル**（レベル帯による変動なし）。
- **累積**: 消費しきれない SP は **`skillPoints` として保持**し続ける。
- **将来拡張**: 実績・クエスト等からの SP 付与は、仕様に明記してから追加可能（現時点ではレベルのみ）。

### 10.6 リスペック（パークの振り直し）

- **方針**: 提案書セクション7の **個別取り消し系（C）**。**通貨での全リセットは採用しない**（仕様の主軸はレアアイテム）。
- **コスト**: 指定操作のたびに **レアアイテムを 1 個消費**する。
  - アイテムの **ID・表示名・入手経路**（ショップ、低確率ドロップ、実績報酬など）は **別途データで定義**する。本仕様では「レアアイテム枠」として位置づけ、実装時に `PlayerData` の所持（例: 消費アイテム配列）へ載せる。
- **対象**: プレイヤーが選んだ **解放済みノード 1 つ**を未解放に戻す。
- **DAG 整合性（下流の連鎖）**: 取り消したノードを `requires` に含む **他の解放済みノード**がある場合、それらも **未解放に戻す**（トランジティブに、整合性が取れるまで）。プレイヤーに「連鎖で外れるノード」と「返却 SP 合計」を**確認 UI で明示**してから確定する。
- **SP 返却**: 取り消し対象ノードおよび連鎖で外れたノードについて、それぞれに支払っていた **コスト SP を全額 `skillPoints` に返却**する。
- **禁止例**: 連鎖結果が不明瞭な状態を作らないよう、**確定時点でグラフが再び有効な DAG** になること（実装は「外す集合」を計算してから一括コミット）。
- **UI**: スキル詳細から「取り消し（レアアイテム使用）」を出す想定。アイテム未所持時は説明付きで disabled。

### 10.7 MVP：5 ツリー × 各 8 ノード（設計）

#### 10.7.1 規模

- **ツリー数**: **5**
- **ノード数**: **各ツリーちょうど 8**（合計 40 ノード）
- **5 本共通の DAG 形**を採用する（実装・UI・検証を単純化）。テーマごとに **名前・効果カテゴリのみ**差し替える。

#### 10.7.2 ツリー一覧（`treeId`・表示名・ステータスとの対応）— **確定**

**方針（Q8）**: 統合 Book の **ステータスタブ**に出ている **「現在の能力値」4項目**に **1:1 で対応するツリーを 4 本**置き、**ステータス指数に載らない独自パーク**を **`special` ツリー 1 本**に集約する。

**根拠（実装参照）**: `GameScene.fillBookStatusPanel` および `data-stat-key`（`power` / `speed` / `control` / `rare-hunt`）と説明文。

| `treeId` | スキルUI表示名（案） | ステータス画面での名称 | 主に伸ばす内容（ステータス算出との関係） |
|----------|----------------------|------------------------|------------------------------------------|
| `power` | パワー | **パワー** | **キャスト距離**系（`rod.castDistanceBonus` など）。遠投・大物補助の方向性。 |
| `speed` | スピード | **スピード** | **ゲージ増加速度**系（竿の catch 補正＋レベル `gaugeSpeed` と同系。スキルは **別バケツで加算**し、合成は `fillBookStatusPanel` と同じ考え方）。 |
| `control` | コントロール | **コントロール** | **バー判定範囲**系（`effectiveBarHeight` と同系。レベル `barRange` と **分離してから加算**し二重計上しない）。 |
| `rare-hunt` | ラック | **ラック** | **レア〜伝説の遭遇**系（エサ・ルアー・竿のレア補正の合成 `rareHuntScore` と同系。スキル分を同指数に反映）。 |
| `special` | 特殊（案） | （ステータスに **専用行なし**） | **`abilityId` を主役**にするツリー。例: 売却・バッグ枠・図鑑・ゴミ率・ファイト内ワンショット救済など **4 指数にマッピングしにくい**もの。数値フィールドも **4 指標以外**に限定するのが望ましい。 |

- **`treeId` はセーブ・ノード ID 接頭辞として固定**する（`rare-hunt` のハイフン含む）。
- スキル画面ではツリー切り替えタブのラベルを、上表の **ステータス名称と揃える**とプレイヤーが対応関係を理解しやすい。

#### 10.7.3 共通トポロジー（8 ノード・DAG）

すべての `treeId` で **同じ前提関係**とする。ノード ID は `{treeId}_n01` … `{treeId}_n08`（実装時の命名はプロジェクト規約に合わせてよいが、**ツリー内で一意**であること）。

| ノード | レイヤー | 前提 `requires` | 役割（全ツリー共通の「位置」の意味） |
|--------|----------|-----------------|--------------------------------------|
| n01 | 序盤・根 | `[]` | ツリー入口 |
| n02 | 序盤 | `[n01]` | 導入強化 |
| n03 | 序盤（分岐手前） | `[n02]` | 中盤分岐の起点 |
| n04 | 中盤・左枝 | `[n03]` | ビルド分岐 A |
| n05 | 中盤・右枝 | `[n03]` | ビルド分岐 B |
| n06 | 中盤・合流 | `[n04, n05]` | **AND 合流**＋**第2特殊アビリティ**（10.10） |
| n07 | 終盤 | `[n06]` | 合流後の **ステータス**強化（極め手前） |
| n08 | 終盤・キャップストーン | `[n07]` | そのツリーの王牌（ステータス） |

これは Q1 で決めた **「中盤まで分岐・終盤は収束・上位は極め」** と整合する。

### 10.8 セーブマイグレーション（スキル導入時）

#### 10.8.1 既存プレイヤーへの初期 `skillPoints`（Q6・**案A**）

- **対象**: スキル機能追加**以前**に作られたセーブで、`skillPoints` フィールドが**無い**（またはマイグレーション未実行とみなす）場合。
- **付与式**: **`skillPoints = 3 × (level − 1)`**（`level` はセーブ上の現在レベル）。
  - 例: Lv10 なら **27 SP**。Lv1 なら **0**。
- **意味**: 「過去に到達した各レベルアップ分、あとからでも SP が 3 ずつ付いていた」扱い（10.5 のレベルアップ付与ルールと整合）。
- **解放済みノード**: マイグレーション時点では **`unlockedSkillNodes` は空**（または未定義なら空で初期化）。**過去のレベル分の SP はすべて未消費で手持ち**となる。

#### 10.8.2 実装ルール（推奨）

- **一度きり**: `skillPoints` が**既に数値として存在する**セーブでは、上式で**上書きしない**（プレイ中の消費状態を壊さない）。
- **新規プレイヤー**: `createInitialPlayerData` で `skillPoints: 0`（Lv1 想定）とし、上記マイグレーション式は不要。
- **`unlockedSkillNodes`**: 未定義なら空集合で初期化。マイグレーション Version フラグ（例: `skillSystemVersion: 1`）を持たせると、将来のルール変更時に安全。

#### 10.8.3 新規フィールドの一覧（参照）

- `skillPoints: number`
- `unlockedSkillNodes: string[]`（または `Set` 相当のシリアライズ）
- （任意）`skillSystemVersion: number`

### 10.9 効果データの表現（Q7・確定）

#### 10.9.1 方針

- 当初の選択肢 **A（列挙フィールド）** と **C（数値＋特殊は `abilityId`）** の **ハイブリッド**を採用する。全面 **B（タグ＋汎用配列）** だけにしない（バリデーションとデバッグの負担を避ける）。
- **狙い**: 解放済みノードを走査して **数値ボーナスを一箇所で合算**しやすくしつつ、「一度だけ救済」など **手続き的な処理**は ID で拡張する。

#### 10.9.2 ノード定義（概念スキーマ）

各 `SkillNode` は概ね次を持つ（実装時の型名は任意）。

- **`effects`（またはノード直下）** に、次の **オプショナルな数値フィールド**を並べる。未定義は 0 または 1（乗算の単位）として扱うルールを `getSkillStatBonuses(playerData)` のような **単一関数**に集約する。
  - 例（名前は実装で確定）: `barRangeAdd`, `gaugeSpeedAdd`, `sellPriceMul`, `rareChanceAdd`, …  
  - **レベルボーナス**（10.4）と同名の概念が出る場合は、**スキル分はここだけ**を読み、既存 `getLevel*Bonus` と **加算前に分離**してから合成する（二重計上防止）。
- **`abilities?: string[]`**: 特殊効果の **`abilityId`**。挙動は **レジストリ**（`Record<abilityId, SkillAbilityHandler>` または switch）に登録し、`GameScene` / インベントリ処理など **必要なシーンでだけ**参照する。

#### 10.9.3 実装メモ

- **MVP**: 数値キーは **本当に使う 1〜2 種類**から始め、フィールドを増やす。
- **`abilityId`**: 新パークを足すときに **ID とフック先をペアで追加**。一覧表を仕様に固定しない（チェックリストは方針レベルで完了とする）。
- **UI 表示**: 数値はフィールドから説明文を生成、`abilityId` は別マップで **表示名・説明**を持たせるとよい。

### 10.10 全ノード定義（コスト・種別・内容案）

#### 10.10.1 ノード種別（アビリティ位置）

- **特殊アビリティ**は **n03（1つ目）** と **n06（2つ目）**。合流ノード n06 で「第2アビリティ」を取る設計。
- **それ以外**（n01, n02, n04, n05, n07, n08）は **ステータス加算**（該当ツリーの数値フィールド。10.9）。

#### 10.10.2 開放コスト `costSp`（確定）

**1〜8 行目**を、それぞれ **n01〜n08** の取得コストとする。7・8 段目（n07・n08）は **3 SP**。

| 段（スロット） | ノード | `costSp` | 種別 |
|----------------|--------|----------|------|
| 1 | n01 | 1 | ステータス |
| 2 | n02 | 2 | ステータス |
| 3 | n03 | 3 | **アビリティ** |
| 4 | n04 | 2 | ステータス |
| 5 | n05 | 2 | ステータス |
| 6 | n06 | 5 | **アビリティ** |
| 7 | n07 | 3 | ステータス |
| 8 | n08 | 3 | ステータス |

- **1 ツリー合計**: \(1+2+3+2+2+5+3+3 = 21\) SP。
- **5 ツリー合計**: \(21 \times 5 = 105\) SP。

#### 10.10.3 全ツリー取得に必要な SP とレベル

- **レベルアップ 1 回あたり +3 SP**（10.5）のみとすると、累積は **`3 × (現在レベル − 1)`**（Lv1 で 0）。
- **必要なレベルアップ回数**: \(\lceil 105 / 3 \rceil = 35\) 回。
- **Lv35** 到達時: \(3 \times 34 = 102\) SP → **3 不足**。
- **Lv36** 到達時: \(3 \times 35 = 105 \ge 105\) → **全ノード分の SP が初めて足りる**。

**要約**: 上記コスト表のとき、SP 入手がレベルのみなら **理論上 Lv36** で 5 ツリー全解放が可能（DAG 前提を満たす取り順で）。

#### 10.10.4 ステータス加算の記法（データ用メモ）

- 表中の **数値**は **スキル専用バケツへの加算（または乗算の加算成分）**の **目安**。実装時に `getSkillStatBonuses` で合算し、既存の装備・レベル式と **二重計上しない**こと（10.4）。
- 略記:
  - **キャスト**: `castDistSkillAdd`（最終的にパワー指数へ）
  - **ゲージ**: `gaugeSpeedSkillAdd`
  - **判定帯**: `barRangeSkillAdd`（0〜1 の範囲内で clamp）
  - **ラック**: `rareHuntSkillAdd`（表示前の合成スコアへの加算成分）
  - **特殊ツリー**: `sellPriceSkillMul`（売却に乗算）、`junkRateSkillMul`（ゴミ率に乗算 1 未満で減少）、`maxInventorySkillAdd`（整数枠、キャップのみ）
